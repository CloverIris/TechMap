## 2. 产品定义

SeekStar TechMap 是一个面向开发者的、可探索的 3D 技术百科与关系图谱。它以宏观景观展示计算机技术处于哪里、由什么构成、依赖什么、运行在哪里、与什么互为替代，以及如何沿时间演化。

它首先是一张地图，而不是技术栈生成器、AI 咨询站、排行榜或营销入口。用户应先看见全局，再逐步进入一个领域、一条路径或一个实体。

### 2.1 核心用户

Alpha 的首要用户是：

- 正在使用 AI 编程工具，但缺少横向技术景观的 Vibe coder。
- 准备制作第一个网站、应用或 AI 产品的新人构建者。
- 希望快速理解陌生技术上下文的开发者、产品设计者和技术学习者。

专业开发者也是受益者，但 Alpha 不以替代专业架构评审、采购评估或安全审计为目标。

### 2.2 核心用户任务

用户必须能够完成以下任务：

1. 在全局地图中找到一个已知或陌生技术，并理解其所在领域。
2. 识别该技术至少一个上游基础、一个直接关联和一个可替代对象（如确有替代关系）。
3. 沿抽象层观察技术如何从硬件、系统、运行时走向框架和产品。
4. 沿时间观察技术、语义时代及生命周期事件的演化。
5. 在概念、组织、人物、论文与时代版本之间切换上下文，而不把它们混成同一种节点。
6. 分享一个可恢复的语义地图状态。

### 2.3 价值主张

核心价值是建立技术世界的可见结构：

- **位置**：它属于哪个学科与能力域。
- **构成**：它建立在哪些技术之上，又承载哪些技术。
- **关系**：它与哪些对象集成、配套、替代、继任或互相启发。
- **历史**：它何时出现、何时形成语义时代、何时改名或结束支持。
- **证据**：用户可以追溯关键事实、来源、审核状态和争议。

“替用户生成一个方案”不是 Alpha 的核心价值，也不属于 Alpha 范围。

## 3. 产品不变量

以下内容在 v1.x 中视为产品不变量：

1. 产品名称为 **SeekStar TechMap / 技术地图**，主域名为 `techmap.seekstar.ai`。
2. 产品的第一屏是可操作地图，不是营销首页、问卷或聊天窗口。
3. 核心对象是有证据支撑的技术关系图谱，3D 是表达结构的主要媒介。
4. Atlas、Stack、Time 分别表达横向版图、抽象层和年代，不得混用语义。
5. 节点或区域的视觉面积不得暗示流行度、质量、商业价值或推荐强度。
6. 事实层与观点层必须分离；观点或生命周期判断只有在有来源时才能显示。
7. Alpha 无需登录即可使用，不建立个人画像，不记录个人地图历史。
8. 正式图谱只能通过人工审核进入；Agent 不得自动合并事实。
9. 核心数据集采用 CC BY 4.0，并携带版本、内容哈希和逐事实证据。
10. 中文体验是发布门槛，数据模型与界面架构从第一天支持 locale。

改变任一不变量都需要决策记录、影响分析和 PRD 版本升级。

## 4. Alpha 成功定义

Alpha 要完成“完整计算机世界的宏观骨架”，并深挖两条高需求纵向链：

- Web 应用链：从硬件、操作系统、网络与浏览器运行时，到 Web 标准、前端框架、后端、数据、部署和产品。
- AI 应用链：从硬件与 CUDA 类计算平台，到框架、模型、推理、数据、应用框架、部署和产品。

硬件到高级语言链作为第三条引导路径，用于验证跨层表达是否成立。

Alpha 不是“百科全书已经收录一切”的宣称。它必须让十二个根领域形成连续、可理解的宏观景观，同时在 Web 与 AI 两条链上达到足以学习和追溯的深度。

### 4.1 内容规模

- 主技术目标为约 300 个，验收区间为 **280 至 320 个**。
- 加入直接维护组织、关键创建者、奠基论文和语义时代版本后，全部实体目标为 **500 至 700 个**。
- 语义时代版本不计入主技术配额。
- 十二个根领域不得为空，也不得仅以占位节点满足覆盖。

“主技术”是地图默认技术景观中的独立实体。是否纳入由解释价值、结构作用、历史影响和当前生态相关性共同决定，不由单一热度指标决定。

### 4.2 发布形态

达到全部门槛后，Alpha 以无需登录的公开实验版发布。所有地图与详情页面必须持续显示：

- 数据版本。
- 数据内容哈希的短形式。
- 布局版本。
- 最近审核时间。
- 当前渲染后端：WebGPU、WebGL2 或基础模式。

## 5. 信息架构

### 5.1 首屏

访问首页后，用户直接进入全屏地图。允许显示一个可跳过的短引导，用于说明三种视图、搜索和聚焦操作，但不得阻塞地图加载，也不得要求用户先回答技术偏好。

首页不得包含：

- 营销式 Hero 页面。
- 登录墙。
- 入站问卷。
- 默认打开的 AI 对话。
- 强制播放的长教程。

### 5.2 核心界面区域

桌面端核心界面由以下部分构成：

- 全屏 3D 地图画布。
- 视图切换：Atlas、Stack、Time。
- 全局搜索与结果定位。
- 类型、领域、年份和叠层筛选。
- 时间滑杆（Time 视图主控件，其他视图可作为筛选器使用）。
- 实体聚焦后的详情抽屉。
- Guided Paths 入口与步骤导航。
- 复位、标准正视角与分享操作。
- 数据、布局、审核和渲染状态栏。

实体必须另有可索引详情页。详情抽屉用于保持地图上下文，详情页用于稳定链接、搜索引擎收录和无 3D 后端时的基本访问。

### 5.3 渐进披露

远景只展示领域、一级地标和必要标签。缩放或聚焦后逐步显示区域、二三级地标、局部节点和关系。默认不得同时绘制全部关系边。

用户的认知顺序应是：

1. 看到领域轮廓。
2. 认出少量地标。
3. 进入一个局部区域。
4. 聚焦实体并查看关系。
5. 打开证据、事件和更完整的详情。

## 6. 十二个根领域

Atlas 的横向版图固定由以下十二个根领域构成：

1. 理论与算法
2. 硬件与体系结构
3. 操作系统与系统软件
4. 语言与运行时
5. 网络与分布式系统
6. 数据与存储
7. 图形媒体与人机交互
8. 应用平台与框架
9. 软件工程与开发工具
10. 云基础设施与运维
11. 安全隐私与身份
12. AI 与智能系统

根领域的 ID、语义和数量在 v1.x 中固定。可以新增子领域，但不得通过改名或拆并根领域绕过本体主版本变更。

技术可同时与多个领域存在分类关系，但地图中的物理包络必须有一个明确的主归属，以保证布局可解释和可确定。跨领域意义通过关系、标签和局部连接表达，不复制同一实体。

## 7. 三种空间视图

三种视图共享同一份图谱事实和选中状态，但使用不同的规范化布局投影。它们不是同一个坐标系的任意相机角度。

### 7.1 Atlas：横向技术版图

Atlas 回答“它在哪里，与哪些领域相邻”。

- X/Y 平面表达十二领域与子区域的横向邻接。
- Z 仅用于浅浮雕、层次提示和选中反馈，不承载年代或抽象层事实。
- 默认接近俯视，允许有限倾角，以保留地图感和地标立体感。
- 领域边界采用正交包络，经视觉柔化后呈现，不使用任意圆形气泡。

### 7.2 Stack：抽象层与构建层级

Stack 回答“它建立在哪一层，又承载哪一层”。

- Z 轴表达公共抽象层带，例如物理硬件、体系结构、系统软件、运行时、平台、框架、应用与产品。
- 抽象层不是严格的全序。实体可以跨越多个局部层带，并通过 `layer_span` 表达跨度。
- X/Y 尽量保持 Atlas 的领域邻接和用户方向感，但允许为关系可读性进行受控局部重排。
- Stack 不得把发布日期直接映射到高度。

### 7.3 Time：技术年代与生命周期

Time 回答“它在什么时候出现和演化”。

- Z 轴表达年代，旧技术在下，新技术在上。
- 主时间锚点使用实体的首次公开发布或最早可证实公开出现时间；精度不足时必须显式显示年份、月份或约数精度。
- 语义时代版本和生命周期事件沿时间轴展开。
- 时间滑杆用于限定可见年份或观察某一历史截面。
- Time 不得把“更高”解释为“更高级”“更好”或“更流行”。

### 7.4 视图切换

视图切换保留焦点、筛选、叠层、路径和可适用的年份状态，通过动画在已发布的布局快照之间过渡。动画结束后必须落在确定性的目标坐标，不得把用户相机浮点值写回公共状态。

每个视图必须提供：

- 复位。
- 标准正视角。
- 当前视图名称及其轴语义。
- 受控缩放和平移。

Atlas 限制倾角；Stack 与 Time 允许受限环绕。所有限制都以防止迷失方向和标签失读为准。

## 8. 地图形态与动态包络

### 8.1 区域

区域代表根领域或经编辑确认的子领域。区域面积只表示当前发布版本中内容物的空间包络，不代表技术重要性、市场份额或生态规模。

区域不得预先写死尺寸。布局管线必须从内容确定性推导区域边界：

1. 根据名称、类型标识和最低可读尺寸计算每个子节点的占地。
2. 按关系、编辑锚点和稳定性约束放置子节点。
3. 计算能包含全部直属子节点的最小正交包络。
4. 为区域标题保留独立、不可被节点侵占的标题带。
5. 加入统一内边距、节点间距和边界缓冲。
6. 解决同级区域重叠，并只重排受影响的局部邻域。
7. 在渲染阶段柔化边角，但不改变命中区域或语义边界。

新增 React、Vue、Next.js 等节点时，Frontend 子区域必须由上述过程扩张并重新排布，不得要求编辑者手工计算新宽高。

### 8.2 包络树与多重分类

为避免循环包络和一个实体同时出现在多个物理位置：

- 每个可见实体在一个布局快照中只有一个 `primary_container_id`。
- 一个容器不得成为自己的祖先，包络树必须无环。
- `member_of`、`is_a` 等图谱关系可以多重存在，但只有主容器关系参与物理包络。
- 跨容器关系通过局部关系边和详情表达，不通过复制节点表达。

### 8.3 技术地标

技术实体使用浅浮雕地标，而不是高楼式体量竞赛。节点面积只服务于：

- 标签能否完整或合理缩略显示。
- 类型图标与交互命中面积。
- 当前缩放层级的可读性。

地标分三级，只控制远景可见性和标签优先级：

- **一级地标**：跨领域基础、关键历史转折或大范围上游依赖。
- **二级地标**：领域内结构性关键技术。
- **三级地标**：局部生态中有解释价值的技术。

每个地标等级必须提供公开的编辑理由；等级不是质量评分、热度榜或采用建议。

### 8.4 颜色、形状与品牌资产

- 颜色主要编码领域。
- 形状或通用图标主要编码实体类型。
- 选中、路径、争议和生命周期使用独立且可辨识的状态编码。
- 不得用节点面积或饱和度表达流行度。
- Logo 仅在来源与商标使用规则明确时使用，并记录资产来源和适用条款。
- 无明确依据时回退到官方文字标识或通用类型图标。

整体风格应接近现代数字地图与地质剖面，不采用星空、科幻驾驶舱或纯装饰性 3D 景观。

## 9. 关系与探索交互

### 9.1 默认关系密度

关系边默认不全量显示。以下情形可以显示关系：

- Guided Path 当前路径。
- 用户聚焦节点的一跳局部邻域。
- 用户明确选择的关系类型。
- 编辑定义的少量跨域地标路径。

关系边必须具有方向、类型和可访问的文字说明。对称关系在数据层仍需明确其对称规则，避免前端猜测。

### 9.2 搜索

Alpha 搜索覆盖：

- 中文名与英文正式名。
- 别名与历史名称。
- 常用缩写。
- 中文拼音。
- 精确匹配、前缀、子串与 trigram 相似度。

排序应优先精确名称和别名，其次前缀、缩写、拼音与模糊匹配。地标等级可以处理同分，但不得把未公开的趋势评分混入排序。

搜索正文不得写入分析日志。Alpha 不引入向量检索、Embedding 搜索或独立搜索服务。

### 9.3 聚焦与详情

聚焦实体后必须显示：

- 正式名称、中文显示名、类型和一句摘要。
- 所在领域与主容器。
- 核心上游、下游、替代、配套和继任关系。
- 关键事件与语义时代。
- 生命周期状态及证据。
- 维护组织、关键创建者和奠基论文（如适用）。
- 来源、最近审核时间、争议状态。
- 打开完整详情页与报告问题的入口。

### 9.4 版本展开

大版本只在形成不同兼容性、生态、架构或历史阶段时建立“语义时代”子节点。默认折叠；用户可在详情或地图中展开。不得为每个补丁或常规版本机械建点。

### 9.5 叠层

概念、标准、硬件、软件与产品默认显示。以下实体默认隐藏，通过可选叠层打开：

- 组织。
- 人物。
- 论文。

观点叠层与事实层分离。Alpha 仅允许显示有来源支持的生命周期状态，不提供无证据的“正在崛起”“已经过时”或趋势评分。

## 10. Guided Paths

Alpha 必须提供三条编辑策划的引导路径：

1. **Web 产品**：从底层执行环境到浏览器/服务器运行时、Web 标准、应用框架、数据、部署与可用产品。
2. **AI 应用**：从计算硬件与驱动/计算平台到机器学习框架、模型与推理、数据与应用层。
3. **硬件到高级语言**：从处理器与指令体系，经操作系统、编译器/虚拟机/运行时到高级语言和框架。

每条路径必须：

- 有明确的学习问题，而不是“最佳技术栈”结论。
- 包含有序步骤、步骤间关系和简短解释。
- 在三视图中尽可能保持上下文。
- 可以中途退出、前后移动和重新开始。
- 由版本化数据定义，不写死在前端组件中。
- 使用与正式图谱相同的实体与 Statement 证据，不复制事实。

## 11. 国际化、路由与分享

### 11.1 Locale

中文 UI 完整可用是 Alpha 发布门槛。系统从第一天使用 locale-aware 路由、文案和数据字段，不允许把中文硬编码进 3D 布局逻辑。

所有实体必须具备：

- 英文正式名称。
- 至少一句英文摘要。
- 已知别名与缩写。

中文本地化名称和摘要应在 Alpha 覆盖全部默认可见实体。技术主显示名优先使用官方名称，例如 React、Vue.js、CUDA，不强制翻译品牌名。

### 11.2 可索引详情页

实体详情使用 locale 与类型分段路由，例如：

- `/zh/tech/{slug}`
- `/zh/org/{slug}`
- `/zh/person/{slug}`
- `/zh/paper/{slug}`

其他实体类型按稳定类型段扩展。Slug 可以迁移并建立重定向；永久 ID 不变且不得复用。详情页必须以服务端可索引 HTML 呈现核心定义、关系、事件和来源，不能只存在于 Canvas 中。

### 11.3 分享状态

分享链接只保存语义状态：

- 视图。
- 焦点实体永久 ID。
- 年份或时间范围。
- 类型与领域筛选。
- 已启用叠层。
- Guided Path 及当前步骤。
- 分享状态格式版本。

链接不得保存相机位置、旋转四元数或其他浮点坐标。打开链接时，由对应布局快照和标准取景规则恢复构图。旧 slug 应重定向，旧状态格式应迁移或给出清晰降级提示。

## 12. Alpha 范围

### 12.1 必须包含

- 无需登录的全屏 3D 地图入口。
- Atlas、Stack、Time 三视图及标准视角。
- 十二根领域和 500 至 700 个总实体的发布数据。
- Web 与 AI 两条深链及第三条硬件到高级语言路径。
- 搜索、定位、筛选、局部关系、叠层和时间滑杆。
- 语义时代版本展开。
- 实体详情抽屉与可索引详情页。
- Statement 级来源、审核时间与争议状态。
- 可恢复的分享链接。
- WebGPU、WebGL2 等价渲染与基础搜索/详情降级模式。
- 中文完整体验和 locale 基础设施。
- 数据版本、内容哈希、布局版本、审核时间和渲染后端展示。

### 12.2 明确不包含

- 账户、登录、云端收藏和跨设备历史。
- Advisor 或聊天式技术咨询。
- 公开 REST/OpenAPI、GraphQL 或 MCP 服务。
- Benchmark 榜单与模型测试平台。
- PWA、完整离线使用或离线数据编辑。
- 趋势、热度、质量或推荐评分。
- 自动生成并自动合并的图谱事实。
- 完整 2D 图谱或与 3D 探索完全等价的无障碍体验。
- 企业私有图谱、团队协作或商业采购功能。

“不包含”项不得以隐藏实验入口、临时接口或未经评审的顺手功能进入 Alpha。

## 13. 数据模型

### 13.1 核心类型

Alpha 数据契约由以下核心类型构成：

| 类型 | 责任 |
| --- | --- |
| `Entity` | 可被命名、引用、检索和展示的对象 |
| `Domain` | 领域及子领域的规范定义与布局锚点 |
| `Relation` | 两个对象之间有类型、方向和有效期的关系 |
| `Event` | 一个或多个对象在某个时间发生的生命周期事实 |
| `Statement` | 将一个可审核断言与证据、状态和审核记录绑定 |
| `Source` | 官方文档、标准、论文或其他证据的规范化引用 |
| `LayoutSnapshot` | 对特定数据版本生成并审核的不可变布局结果 |
| `GuidedPath` | 使用正式实体和关系定义的编辑路径 |

这些类型的机器可读 Schema 由未来的公共 `techmap-data` 仓库发布。产品仓不得维护一套语义不同的影子 Schema。

### 13.2 永久 ID

所有对象使用小写、可读、带命名空间的永久 ID，例如：

- `tech.react`
- `org.meta`
- `person.brendan-eich`
- `paper.attention-is-all-you-need`
- `domain.ai`
- `event.react-18-release`

ID 一经公开不得改写或复用。名称、归属、slug 和维护组织可以变化；变化通过别名、重定向、事件与有效期表达。若实体被误建，应保留 tombstone 或合并映射，不得让旧 ID 指向另一个对象。

### 13.3 Entity 类型

`Entity.type` 冻结覆盖：

- `domain`
- `concept`
- `algorithm`
- `standard`
- `hardware`
- `architecture`
- `system`
- `language`
- `runtime`
- `protocol`
- `library`
- `framework`
- `database`
- `tool`
- `platform`
- `service`
- `product`
- `organization`
- `person`
- `paper`
- `era_version`

新增核心类型属于本体变更，必须通过决策记录。领域特有的细分优先使用标签或版本化扩展命名空间。

### 13.4 Entity 最低字段

正式发布的 Entity 至少包含：

- 永久 `id` 与 `type`。
- 英文正式名称。
- locale 化显示名称、短摘要和可选长说明引用。
- 别名、缩写和历史名称。
- 当前 slug 与 slug 迁移记录。
- 主领域与主容器。
- 首次公开时间锚点及精度（如果该实体具有发布时间）。
- 地标等级及公开理由（如为地图地标）。
- 最近审核时间、审核策略和内容状态。
- 支撑定义与关键字段的 Statement 引用。

### 13.5 Domain 与主容器

`Domain` 除本地化内容外，还定义：

- 根领域或父领域 ID。
- 稳定排序键。
- Atlas 编辑锚点。
- Stack 默认层带或跨度。
- 布局标题与最小留白规则。

实体可以通过关系归入多个领域，但 `primary_domain_id` 和每个布局中的 `primary_container_id` 必须唯一。主容器只解决物理布局，不宣称唯一学术分类。

### 13.6 语义时代版本

`era_version` 是从属于主技术的折叠子节点。只有满足至少一个条件时才建立：

- 引入明显的兼容性断点。
- 改变核心架构或运行模型。
- 形成新的生态阶段。
- 对技术史具有独立解释价值。

它必须带有效期或起始事件、与主实体的关系和来源。常规小版本不得建点。

## 14. 关系本体

### 14.1 核心关系词表

v1 核心关系机器名冻结如下：

| 机器名 | 中文语义 | 方向示例 |
| --- | --- | --- |
| `is_a` | 分类为 | React `is_a` UI library |
| `member_of` | 归属于 | React `member_of` Frontend |
| `built_on` | 构建于 | Next.js `built_on` React |
| `runs_on` | 运行于 | Node.js `runs_on` V8 |
| `compiles_to` | 编译到 | TypeScript `compiles_to` JavaScript |
| `implements` | 实现 | Chromium `implements` Web standards |
| `targets` | 面向/目标为 | CUDA compiler `targets` NVIDIA GPU |
| `integrates_with` | 集成 | Framework `integrates_with` Database |
| `complements` | 配套互补 | Tool `complements` Framework |
| `alternative_to` | 可替代 | Vue `alternative_to` React |
| `successor_to` | 继任 | Technology B `successor_to` Technology A |
| `inspired_by` | 受启发于 | Technology B `inspired_by` Technology A |
| `maintained_by` | 由其维护 | React `maintained_by` Meta |
| `created_by` | 由其创建 | Technology `created_by` Person/Organization |
| `used_for` | 用于 | Technology `used_for` Capability/Concept |
| `introduced_in` | 由论文引入 | Algorithm `introduced_in` Paper |
| `authored_by` | 论文作者为 | Paper `authored_by` Person |
| `published_by` | 发布组织为 | Paper/Standard `published_by` Organization |

关系方向由 Schema 规定，前端不得根据中文标签猜测。`alternative_to` 与 `complements` 可在展示层双向呈现，但数据必须遵守去重和规范排序规则。其他关系不得自动反转成同义事实。

### 14.2 扩展关系

领域特有关系先进入带版本的扩展命名空间，例如 `ext.ml.v1.trained_with`。扩展关系必须声明：

- 定义、方向和允许的端点类型。
- 是否对称、是否传递。
- 有效期和版本范围规则。
- 映射到核心关系的可能性。
- 至少一个验证样例。

扩展关系不能以自由文本标签混入核心数据。提升为核心关系需要本体决策记录和版本评估。

### 14.3 图规则

- 主容器树、`successor_to` 和语义时代顺序必须无环。
- `built_on`、`runs_on` 等事实图可能因抽象或互操作产生复杂结构，校验器必须使用关系特定规则，不能粗暴禁止全部循环。
- 每条关系必须有唯一 ID、方向、端点、Statement、可选版本范围和有效期。
- 不确定或有争议的关系不得伪装成已接受事实。

## 15. 时间、事件与有效期

### 15.1 时间表达

日期必须携带精度，至少支持：

- `day`
- `month`
- `year`
- `circa`
- `range`
- `unknown`

不得把只有年份的来源补成虚假的月日。多个可信来源冲突时，Statement 必须保留争议并说明采用的显示值。

### 15.2 核心事件类型

生命周期由事件表达，核心事件覆盖：

- `announced`
- `first_public_release`
- `stable_release`
- `major_release`
- `renamed`
- `ownership_changed`
- `deprecated`
- `archived`
- `end_of_support`

`major_release` 只用于具有地图解释价值的语义时代起点。普通发布可以留在长说明或外部版本资料中。

### 15.3 有效期与版本范围

关系和事件必须支持：

- 生效开始与结束时间。
- 时间精度。
- 适用的技术版本或语义时代。
- 当前是否仍有效。

例如维护组织变化不得覆盖旧事实；应以两个具有不同有效期的 `maintained_by` 关系和一个 `ownership_changed` 事件表达。

## 16. Statement 与来源

### 16.1 逐事实证据

`Statement` 是正式图谱进入发布状态的证据单元。它可以断言：

- Entity 的定义、类型或关键字段。
- Relation 的存在、方向、版本范围或有效期。
- Event 的时间、参与者或性质。
- 地标理由或生命周期显示状态。

Statement 至少包含断言目标、状态、来源列表、创建与审核记录、最近审核时间。状态至少区分 `candidate`、`accepted`、`disputed` 和 `retracted`。

### 16.2 来源优先级

来源优先顺序为：

1. 官方项目文档、发布记录、仓库和组织公告。
2. 标准组织规范与正式记录。
3. 原始论文、作者资料与会议出版物。
4. 可信的一手访谈或机构档案。
5. 高质量二手资料，仅用于补足上下文或记录争议。

正式 Alpha 必须达到：

- **100%** 实体定义有至少一个来源。
- **100%** 关键时间有至少一个来源并标记日期精度。
- **100%** 核心关系有至少一个 accepted Statement。
- 全部已接受 Statement 中，至少 **80%** 的主要来源来自官方项目、标准组织或原始出版机构。

“主要来源”是 Statement 用于支持其核心断言的首要证据，不以重复添加官方链接提高比例。

### 16.3 Source 最低字段

Source 至少包含：

- 永久 ID。
- 标题、发布者和 URL 或持久标识符。
- 来源类型与语言。
- 发布或更新时间（如可得）。
- 访问时间。
- 可选归档 URL、版本、许可与内容哈希。

失效链接不得直接删除；应补充归档或替代来源，并保留变更记录。

### 16.4 争议与透明度

存在可靠来源冲突时：

- 显示 `disputed` 状态。
- 保留不同断言与各自来源。
- 记录编辑讨论链接和主维护者裁决。
- 不得让 Agent 用置信度分数替代编辑判断。

实体详情必须能进入来源和讨论记录。纠错入口应预填实体永久 ID、数据版本和当前页面。

## 17. 内容生产与治理

### 17.1 编辑源与运行时

- YAML 保存实体、关系、事件、Statement、Source、路径和布局配置等结构化编辑源。
- Markdown 保存需要版本控制的长说明。
- PostgreSQL 保存产品运行时投影，采用属性图式表结构和逐事实证据关联。
- Drizzle 管理常规 Schema 与查询；复杂图查询可使用经过评审的原生 SQL。

PostgreSQL 是运行时投影，不是绕过公共编辑流程的第二真源。正式数据必须能够从版本化编辑源重建。

### 17.2 仓库边界

未来建立两个清晰边界：

- 公共 `techmap-data` 仓库：数据、Schema、校验器、布局输入规范和贡献流程。
- 私有产品 monorepo：Web、API、布局发布消费、部署和内部运营工具。

产品仓只能消费 `techmap-data` 的版本化发布及内容哈希，不得静默修改已发布数据。数据修正应回到公共仓库，经审核后发布新版本。

### 17.3 贡献与 Agent

公共数据通过 GitHub Pull Request 接受贡献。Agent 可以：

- 搜集候选来源。
- 生成候选 Entity、Relation、Event 或 Statement。
- 运行 Schema、证据和布局影响校验。
- 生成差异摘要与冲突报告。

Agent 不得：

- 自动合并正式图谱。
- 在缺少来源时把推断写成事实。
- 悄悄改变永久 ID、关系方向或证据状态。
- 用模型输出本身作为事实来源。

主维护者拥有最终产品和编辑裁决权。讨论、来源、审核人和争议状态应公开可追溯。

### 17.4 复核周期

- 活跃框架、平台与服务：每 90 天复核。
- 稳定基础技术、标准和历史实体：每 365 天复核。
- 已归档技术：由新事件、来源失效或争议触发复核。

超期不自动删除实体，但必须显示审核过期状态，并进入维护队列。

### 17.5 许可与商标

- 核心数据集采用 **CC BY 4.0**。
- 发布包必须包含语义版本、内容哈希、归属说明和许可证文本。
- 第三方 Logo、商标和受限内容不因数据集许可而自动获得再许可。
- 每个品牌资产必须记录来源、商标规则和使用范围；不明确时使用文字或通用类型图标。

## 18. 布局系统与快照

### 18.1 确定性布局管线

布局由内容 CI 生成，组合：

- 自定义正交矩形包络与标题留白算法。
- ELK 或等价的确定性分层关系布局。
- 人工地标锚点和经审核的钉点。
- 只影响必要邻域的局部重排。

给定相同数据版本、配置、字体度量和算法版本，输出必须逐字节或在规范化精度内一致。禁止在客户端以随机力导向结果作为发布布局。

### 18.2 布局稳定性

- 已钉住的一级地标在非相关内容变更后不得漂移超过规定容差。
- 新增局部节点应优先扩展当前容器与相邻缓冲区。
- 全局重排必须有显式布局版本升级、差异预览和人工批准。
- 标签、标题带和交互命中区不得重叠到不可读。

具体数值容差由首个布局实现通过决策记录冻结，但不得降低本 PRD 的可读性与确定性要求。

### 18.3 LayoutSnapshot

每个已发布 `LayoutSnapshot` 至少绑定：

- 数据版本和内容哈希。
- 布局版本、算法版本和配置哈希。
- 视图类型。
- 节点位置、占地、层带和容器包络。
- 编辑锚点与钉点。
- 标准取景规则与世界边界。
- 生成时间、审核状态与审核人。

审核通过的快照不可变。修正必须发布新布局版本。客户端不得把临时拖拽结果写回公共快照。

## 19. 技术基线

本节冻结未来工程实现的默认技术基线；当前文档阶段不得因此自动创建应用或安装依赖。

### 19.1 产品 monorepo

- pnpm workspace monorepo。
- Node.js 24 LTS。
- TypeScript 6，开启 strict。
- ESLint 与 Prettier。
- 代码标识符、Schema、代码注释和提交消息使用英文。
- 产品与内容文档中文优先，必要时提供英文版本。

安全更新即时处理；普通依赖按月集中更新。Next.js、Three.js 等核心依赖的升级必须使用独立 PR，并执行完整视觉回归。

### 19.2 Web

- Next.js App Router。
- React。
- React Three Fiber 与 Three.js。
- Tailwind CSS。
- Radix UI。
- Lucide 图标。
- Zustand 管理高频交互与地图 UI 状态。
- TanStack Query 管理服务端数据状态。

React Three Fiber 必须通过异步 `gl` 创建 Three.js `WebGPURenderer`。WebGPU 为默认渲染后端；不可用时自动降级至 WebGL2。Alpha 的材质、灯光、后处理和交互必须限定在两种后端都能达到严格视觉与功能等价的共同能力集内，不得以 WebGPU 专属效果制造不同产品体验。

实现依据：

- [React Three Fiber Canvas 文档](https://r3f.docs.pmnd.rs/api/canvas)
- [Three.js WebGPURenderer 文档](https://threejs.org/manual/en/webgpurenderer)

### 19.3 API 与搜索

- Fastify。
- REST/OpenAPI。
- Drizzle ORM 与必要的原生 SQL。
- PostgreSQL。

Alpha 内部接口不承诺外部兼容性，不得被宣传为公共 API。搜索基于 PostgreSQL 的规范化字段、前缀、子串、拼音和 trigram 排序完成，不引入独立搜索集群。

### 19.4 部署与分析

- 首期自托管在单台 Linux 主机，以 Docker Compose 编排。
- CI 构建、测试并扫描 OCI 镜像。
- 服务器只拉取批准的不可变镜像，不在生产机从源码构建。
- 不给予 CI 广泛的生产 SSH 操作能力。
- 使用自托管的隐私分析。
- 只采集匿名、最小化的交互事件。
- 不记录搜索正文、个人地图历史或可重建个人兴趣轨迹的数据。

### 19.5 缓存与离线

Alpha 可以缓存静态资源和最新发布布局快照，但不承诺 PWA、完整离线体验或离线编辑。缓存失效必须绑定数据与布局版本，避免混合不同版本。

## 20. 设备、浏览器与降级

### 20.1 正式支持

正式支持以下浏览器的当前主版本与前一主版本：

- Chrome
- Edge
- Firefox
- Safari

主要性能目标设备为 1080p、中端集成显卡的桌面或笔记本。移动端提供受控 2.5D 构图与基本任务支持，不要求与桌面自由视角完全一致。

### 20.2 渲染降级

渲染能力按以下顺序选择：

1. WebGPU。
2. WebGL2。
3. 基础搜索与详情模式。

WebGL2 模式必须与 WebGPU 模式功能和信息表达等价。允许性能策略不同，例如更保守的像素比、阴影或抗锯齿设置，但截图与可读性必须在已批准容差内。

设备不支持 WebGL2 时，用户仍必须能够搜索实体、阅读详情、关系、事件和来源。Alpha 不要求基础模式提供完整的 2D 图谱等价探索。

### 20.3 Alpha 无障碍基线

- 画布外的导航、搜索、筛选、详情与路径控件使用语义化 HTML。
- 核心控件可由键盘操作并具有可见焦点。
- 尊重减少动态效果偏好。
- 状态不得只依赖颜色表达。
- 基础搜索与详情模式承担无 3D 能力时的核心阅读任务。

完整 2D 图谱及与 3D 探索的 WCAG 任务等价不属于 Alpha 发布门槛，但不得阻止后续实现。

## 21. 性能与体验门槛

### 21.1 加载

在 10 Mbps、稳定网络条件下，以支持设备冷启动测试，从导航开始到用户可以平移、缩放、搜索或聚焦地图必须不超过 **6 秒**。非首屏叠层、长说明和高细节资产应渐进加载。

### 21.2 帧率

在 1080p、中端集成显卡和 Alpha 标准数据集上，执行规定交互脚本时，中位帧率不得低于 **40 FPS**。规定脚本至少覆盖：

- Atlas 平移与缩放。
- 聚焦节点并显示局部关系。
- Atlas 到 Stack、Stack 到 Time 的过渡。
- Time 滑杆变化。
- Guided Path 连续三个步骤。

性能优化不得删除事实、改变关系语义或造成 WebGPU/WebGL2 信息不等价。可使用分级细节、实例化、文字图集、视锥裁剪和渐进加载。

### 21.3 构图稳定

动态标签、Hover、加载文字、版本展开和侧栏开合不得导致地图容器或固定工具栏尺寸跳动。所有固定格式控件必须有稳定尺寸和响应式约束。

## 22. 测试与验收

### 22.1 用户理解测试

至少邀请 12 名目标用户完成可观察测试，其中至少 80% 必须在两分钟内：

1. 找到一个此前不熟悉的技术。
2. 正确识别一个上游关系。
3. 正确识别一个替代项；若该实体没有替代关系，则能理解页面为何未展示替代项。

测试不得由引导者直接告诉用户按钮位置或关系答案。结果需记录任务完成率、用时、误解点和改进决定。

### 22.2 数据与 Schema 自动化

必须覆盖：

- Schema 合法性与永久 ID 唯一性。
- 关系端点类型和方向。
- 对称关系规范化与重复检查。
- 主容器、继任和时代顺序循环规则。
- 来源完整性、官方来源比例和审核时效。
- 日期精度、有效期和版本范围。
- slug 唯一性、迁移与重定向。
- 数据语义版本与内容哈希复现。

### 22.3 布局自动化

必须覆盖：

- 相同输入的布局确定性。
- 容器完整包住直属子节点。
- 区域标题带不被节点侵占。
- 节点、标签与区域的碰撞规则。
- 人工锚点稳定性。
- 局部新增对非相关区域的位移容差。
- 分享状态在目标布局上的恢复。

### 22.4 产品端到端测试

Playwright 必须覆盖：

- Atlas、Stack、Time 三视图切换。
- 搜索并定位实体。
- 语义时代展开与折叠。
- 组织、人物和论文叠层。
- Time 滑杆。
- 三条 Guided Paths。
- 详情页、来源和纠错入口。
- 分享链接恢复。
- WebGL2 与基础模式降级。

### 22.5 视觉与 Canvas 验证

桌面与移动视口都必须执行截图检查。Canvas 测试不能只断言 DOM 存在，还必须验证：

- 非空像素比例，防止黑屏或透明画布。
- 世界边界与首屏构图范围。
- 相机或场景在交互后确实运动。
- 关键标签和工具栏不重叠。
- 目标资源正确加载。
- WebGPU 与 WebGL2 的视觉差异在批准容差内。

Three.js 或渲染依赖升级必须重新运行两种后端的截图与像素验证。

### 22.6 必备数据样例

测试数据和发布数据都必须能验证：

- React、Vue、Next.js 的分类、构建与替代关系。
- Chrome、V8、Node.js 的产品、运行时和运行关系。
- CUDA、TensorFlow、PyTorch 的硬件计算到 AI 框架链。
- 组织、人物和论文叠层。
- 至少一项技术的多个语义时代展开。

样例中的具体日期和关系仍必须经过来源审核，不能把本段举例当作事实证据。

## 23. 发布门槛

Alpha 公开前必须同时满足：

- 十二领域宏观骨架和内容数量门槛。
- 证据覆盖与官方来源比例门槛。
- 三条 Guided Paths 完整且通过编辑审核。
- 12 人用户理解测试及 80% 成功率。
- 10 Mbps 六秒可操作门槛。
- 1080p 中端集显 40 FPS 中位数门槛。
- 四类浏览器的当前与前一主版本验证。
- WebGPU/WebGL2 严格等价验证与基础模式验证。
- 自动化、Playwright、视觉和 Canvas 验证全部通过。
- 中文 UI 与默认可见实体中文内容完整。
- 数据许可、Logo 来源、版本、哈希和审核状态可见。
- 隐私分析经过事件清单审查。

首发不绑定固定日期。未满足任一门槛时保持内部或受限预览状态，不以“实验版”为由跳过证据、隐私或兼容性要求。

## 24. 运营指标

Alpha 指标用于验证地图是否帮助理解，不用于建立个人画像。允许采集的匿名聚合事件包括：

- 地图达到可操作状态的耗时区间。
- 视图切换次数。
- 搜索是否产生结果，不含搜索正文。
- 聚焦、关系展开、版本展开和来源打开次数。
- Guided Path 开始、步骤推进和完成。
- 渲染后端、粗粒度设备能力与错误类型。

不得采集：

- 搜索原文。
- 可串联的个人地图浏览历史。
- 用于广告或跨站跟踪的标识符。
- 未经明确必要性评审的完整相机轨迹。

核心成功信号仍是用户理解测试，而不是停留时长或点击量。

## 25. 风险与约束

| 风险 | 约束与缓解 |
| --- | --- |
| 3D 新奇感压过知识结构 | 三轴语义固定，首屏渐进披露，用户测试以理解任务验收 |
| 数据分类天然多义 | 图谱允许多重分类，物理布局只选一个主容器，并公开关系 |
| 区域增长造成全图漂移 | 确定性包络、人工锚点、局部重排、布局快照审核 |
| 关系过密不可读 | 默认隐藏全量边，仅显示焦点邻域和 Guided Path |
| 历史日期并不精确 | 日期精度是一等字段，不伪造月日 |
| AI 生成事实污染数据 | Agent 只提候选 PR，Statement 证据和人工审核是发布门槛 |
| WebGPU 兼容性不均 | WebGL2 严格等价，无法使用 WebGL2 时进入基础模式 |
| Logo 与商标风险 | 逐资产记录来源和规则，不明确时回退文字/通用图标 |
| Alpha 内容范围失控 | 300 主技术与 500 至 700 总实体，远期能力明确排除 |
| 排行与咨询反客为主 | Advisor、API/MCP、Benchmark 均为非规范远期附录 |

## 26. 变更治理

以下变化必须建立决策记录：

- 产品定位、品牌、首屏或三视图语义变化。
- 十二根领域的新增、删除、改名或拆并。
- 核心 Entity 类型或 Relation 词表的主版本变化。
- 永久 ID、路由或分享状态等公共契约变化。
- 数据许可、来源比例或人工审核门槛变化。
- Alpha 范围新增或删除。
- WebGPU/WebGL2 等价原则、性能门槛或浏览器范围变化。
- 公共与私有仓库边界变化。

决策记录必须说明背景、备选方案、影响、迁移、验证方式和所需 PRD 版本。主维护者拥有最终裁决权。决策被接受后，应先更新 PRD 或明确其目标版本，再进入实现。

小型文案修正、来源补充和不改变语义的澄清可以在 v1.x 内直接修订，但必须保留版本历史。

# 附录 A：非规范远期方向

本附录用于保持长期接口方向一致，不构成 Alpha 需求，不得据此提前扩张 Alpha。

## A.1 Advisor

未来 Advisor 可以作为地图内的上下文抽屉，读取用户当前焦点、路径、筛选和正式图谱，回答“我可以用什么”或比较方案。它应使用多供应商模型适配层，并把事实引用回 Statement 与 Source。

Advisor 不得取代地图首屏，也不得把模型建议写回事实层。

## A.2 公共 REST/OpenAPI 与 MCP

地图成熟后可以发布公共 REST/OpenAPI 与 MCP。届时：

- 使用 `/v1` 路径主版本。
- 只暴露已版本化、可归属的数据。
- 返回 Statement、Source、数据版本和内容哈希。
- 建立速率、弃用、缓存和兼容策略。

Alpha 内部接口没有兼容承诺，也不能被第三方视为预览版公共契约。

## A.3 Benchmark

未来 Benchmark 分为两条独立榜单：

- 模型 API 榜：测试基础模型对技术事实、文档和编码任务的支持。
- 完整 Agent/产品榜：测试包含检索、工具、执行环境和工作流的完整系统。

可考虑的非规范类型：

| 类型 | 作用 |
| --- | --- |
| `BenchmarkSuite` | 版本化任务集合与适用技术范围 |
| `BenchmarkCase` | 单个测试输入、环境、评分器与披露级别 |
| `ModelSnapshot` | 被测模型、供应商、固定版本与参数 |
| `AgentSnapshot` | Agent 产品、工具、系统提示与环境快照 |
| `BenchmarkRun` | 一次可复现执行及完整轨迹引用 |
| `BenchmarkScore` | 分项得分、置信区间、成本与延迟 |

协议、评分器、公开题和可公开轨迹应开放；保留轮换隐藏题以降低过拟合。节奏可采用月度全量榜加新模型发布时的事件测试。Benchmark 不能改变核心地图中的节点面积、领域归属或事实状态。

# 附录 B：冻结术语

| 术语 | 含义 |
| --- | --- |
| 主技术 | 默认技术景观中的独立技术实体，计入约 300 配额 |
| 语义时代 | 具有独立兼容性、架构、生态或历史意义的大版本阶段 |
| 主容器 | 一个布局快照中用于物理包络的唯一父容器 |
| 地标 | 在某缩放层级优先可见的节点，不表示质量或热度 |
| Statement | 带来源、状态与审核记录的单项断言 |
| 布局快照 | 与数据版本绑定、经审核、不可变的确定性空间结果 |
| 基础模式 | 无 WebGL2 时仍可用的搜索与详情体验，不是完整 2D 地图 |
| 观点叠层 | 与事实层分离的解释性显示；Alpha 只允许有来源的生命周期状态 |
